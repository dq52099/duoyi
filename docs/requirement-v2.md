# 多仪需求规格 v2

本文档在 v1（R1-R10）基础上，更新各需求的实现状态，并基于竞品对标分析
新增 R11-R17。目标是将多仪从「功能完备的个人效率工具」推向「智能化、社交化、
国际化的全平台时间管理系统」。

---

## 1. 背景

- 待办：四象限、清单、子任务、重复、截止日、多次提醒（ReminderPlan/ReminderRule）、优先级、标签、AI 拆解子任务。

- 其他：全局搜索、快速捕获 FAB（待办/笔记/日记/AI 创建）、农历、倒数日、纪念日、课程表、笔记、日记、备份导出。

### 1.3 v2 新增对标缺口（竞品分析）

基于对滴答清单、Forest、Habitica、Notion、日事清等产品的竞品分析，新增：

- 智能日期解析：自然语言输入自动解析日期/时间/提醒（对标滴答清单）。
- 看板视图：除四象限外，支持看板拖拽管理任务（对标 Notion/Trello）。
- 专注模式增强：DND 联动、专注社交、更丰富的场景音、标签分类统计（对标 Forest）。
- 习惯追踪增强：模板库扩展、分组分类、弹性打卡（对标 Habitica/Streaks）。
- 数据可视化增强：年度报告、趋势分析、多维交叉分析（对标日事清/滴答清单）。
- 快捷操作：通知栏快捷添加、小组件快捷添加、快捷模板（对标各主流效率应用）。
- 国际化支持：多语言、时区自动处理（面向海外用户拓展）。

## 2. 产品目标

1. 多仪应成为一个「可落地执行」的个人时间系统，而不是只记录任务。
2. 所有提醒必须可验证、可诊断、可重新调度。
3. 所有跨模块事项必须能在日历中统一查看、跳转、编辑。
4. 核心数据必须支持多人共享和冲突处理，同时保留离线可用。
5. 时间统计必须来自真实时间记录，而不是只用完成数量推断。
6. **[v2 新增]** 输入效率：自然语言和快捷操作应让「从想法到记录」的路径尽可能短。
7. **[v2 新增]** 视觉洞察：数据可视化应帮助用户理解自己的时间使用模式和长期趋势。
8. **[v2 新增]** 国际化：应用应对中文和英文用户提供一致体验。

### R1 日历枢纽

> **v2 状态：已部分实现。** CalendarAggregator 已聚合全部 9 种事件类型（todo、
> habit、pomodoro、anniversary、course、diary、countdown、goal、timeEntry），
> 已实现冲突检测和标记（hasConflict/conflictCount），CalendarEventSheet 已提供
> 统一详情面板。尚未完成：日历内直接改期、跨模块编辑回写。

**用户故事**

用户希望在日历里看到今天所有待办、习惯、课程、目标、倒数日、纪念日和日记，
并能直接处理，不需要在多个模块间来回找。

**功能要求**

- R1.1 日历应聚合所有模块事件，统一为 `CalendarEvent` 展示。 **[已实现]**
- R1.2 日/周/月视图都应支持按类型筛选。 **[部分实现]**
- R1.3 点击任一事件应打开统一详情面板，提供跳转、完成、改期、编辑、删除等动作。 **[部分实现 - 跳转和查看已有，改期/编辑待补]**
- R1.4 当同一时间段存在课程、待办提醒、目标提醒时，应提示冲突。 **[已实现]**
- R1.5 今日页和日历页应共享同一套事件排序规则。 **[已实现]**
- R1.6 修改日历中的截止日/提醒时间后，应同步写回原模块 provider。 **[待实现]**

**验收标准**

- 新建一条带截止日期和提醒的待办后，日历日视图、周视图、月视图均可见。
- 在日历中改期后，待办详情页显示新日期。
- 删除原待办后，日历事件消失。
- 冲突事件至少以视觉标记和详情提示展示。

### R2 多次提醒与稍后提醒

> **v2 状态：已基本实现。** ReminderPlan + ReminderRule 模型已落地，支持
> absolute/relativeToDue/dailyTime/weeklyTime 四种规则类型，push/alarm 双通道
> 路由已完成，ReminderScheduler 实现幂等重放，legacy 提醒自动迁移。
> 尚未完成：稍后提醒（snooze）交互。

**用户故事**

用户希望一个任务可以提前一天、提前一小时、到期时分别提醒；错过后可以稍后再提醒。

**功能要求**

- R2.1 `ReminderConfig` 应支持多条 `ReminderRule`。 **[已实现 - ReminderPlan 含 rules 列表]**
- R2.2 ReminderRule 支持绝对时间、相对截止时间、每日固定时间、周几固定时间。 **[已实现 - 4 种 ReminderRuleType]**
- R2.3 每条 rule 可选择 push 或 alarm。 **[已实现 - ReminderKind.push/alarm]**
- R2.4 闹钟提醒支持全屏、震动、稍后提醒。 **[部分实现 - 全屏已有，稍后提醒待补]**
- R2.5 提醒调度必须可幂等重放，避免重复注册。 **[已实现 - ReminderScheduler.resyncAll]**
- R2.6 切换 push/alarm 后必须清理旧通道残留。 **[已实现 - 双通道清理逻辑]**
- R2.7 通知点击应能跳转到对应任务/目标/习惯详情，而不仅是 Tab。 **[已实现 - duoyi:// 深链]**
- R2.8 失败的单条提醒不得中断整轮调度。 **[已实现 - try/catch 隔离]**

**验收标准**

- 一个待办设置 3 条提醒后，pending notification 中存在 3 条可追踪 id。
- 修改其中一条规则后，只保留最新规则对应的调度。
- 关闭通知权限后 UI 显示异常状态；重新授权回到前台后自动刷新并重排。
- Android 精准闹钟权限缺失时，至少退化为普通提醒，并在 UI 引导用户修复。

- R3.1 偏好设置提供通知健康检查页。 **[已实现]**
- R3.2 检查系统通知权限、精准闹钟权限、通知渠道状态。 **[已实现]**

- R3.4 小米/MIUI 设备显示自启动、后台限制、锁屏通知、电池优化的人工检查清单。 **[已实现 - isXiaomiLike 判定 + 4 项手动检查]**
- R3.5 发送测试通知后记录结果和时间。 **[已实现]**
- R3.6 从系统设置返回时刷新权限并重排提醒。 **[已实现]**

**验收标准**

- 未授权通知时，提醒编辑页显示未授权并提供设置跳转。
- 授权后返回应用，状态自动变为已授权。
- 小米设备显示专门的配置说明。
- 测试通知触发后，通知记录页新增一条记录。

### R11 智能日期解析

**用户故事**

用户希望输入「明天下午3点开会」后，应用自动将其解析为一条待办，截止日期设为明天
15:00，并自动添加一条到期提醒，不需要手动选择日期和时间。

**功能要求**

- R11.1 新增 `SmartDateParser` 服务，支持中文自然语言日期时间解析。
  - 支持绝对日期：「12月25日」「2026年1月1日」「下周三」。
  - 支持相对日期：「明天」「后天」「下周一」「三天后」。
  - 支持时间：「下午3点」「上午10:30」「晚上8点半」。
  - 支持组合：「明天下午3点」「下周五上午9点」。
  - 支持语义提取：「明天下午3点开会」→ 标题「开会」、截止日期明天 15:00。
- R11.2 快速捕获 FAB 和待办创建页集成智能解析，输入框底部实时预览解析结果。
- R11.3 解析结果可一键确认或手动修正，避免误解析。
- R11.4 解析引擎应纯本地运行，不依赖网络（离线可用）。
- R11.5 可选接入 AI 服务增强复杂句式解析（如「每周二四的英语课」→ 重复规则）。
- R11.6 英文自然语言解析作为 R17 国际化的配套需求，随 R17 一起交付。

**验收标准**

- 输入「明天下午3点开会」，自动填充标题「开会」和截止日期为明天 15:00。
- 输入「后天」，仅填充截止日期，标题保持用户输入。
- 输入不含日期信息的文本（如「买牛奶」），不产生误解析，保持原样。
- 离线环境下解析功能正常工作。

用户希望专注时手机自动进入免打扰模式，避免被社交消息打断；专注结束后能看到

- R13.5 **[可选]** 专注排行/社交功能：用户可加入专注小组，查看本周专注排行榜。

**验收标准**

### R14 习惯追踪增强

用户希望习惯可以按分类管理（如健康、学习、生活），支持「一周至少跑步 5 次」

### R17 国际化支持

> **v2 状态：已有时区基础。** LocalTimezoneResolver 已实现 IANA 时区自动解析和
> 刷新（含插件探测 → Dart 回退 → Asia/Shanghai 兜底三级策略），
> 但应用内所有文案均为硬编码中文，无多语言支持。

**用户故事**

海外华人或外国友人希望使用多仪，但应用内全是中文，无法理解界面含义。
用户希望可以切换为英文界面，且应用能根据系统语言自动选择。

**功能要求**

- R17.1 接入 Flutter 官方国际化方案（`flutter_localizations` + `intl` + ARB 文件）。
  - 初始支持简体中文（zh_CN）和英文（en）。
  - 默认语言跟随系统 locale，可在偏好设置中手动切换。
- R17.2 所有用户可见的硬编码中文字符串迁移至 ARB 文件。
  - 包含但不限于：导航标签、页面标题、按钮文案、提示信息、空状态文案、成就名称/描述。
  - CalendarEventType.label、TimeEntrySource.label、TimeEntryCategory.label 等枚举 label 迁移。
- R17.3 日期/时间格式国际化。
  - 使用 `intl.DateFormat` 替代手动格式化。
  - 中文：2026年5月15日 周四 15:00。
  - 英文：May 15, 2026 Thu 3:00 PM。
- R17.4 时区自动处理增强（在现有 LocalTimezoneResolver 基础上）。
  - 跨时区旅行时，所有提醒自动适配新时区。
  - 时区变化时触发 ReminderScheduler.resyncAll（已有基础）。
  - 日历事件显示的时间应使用本地时区。
- R17.5 农历功能在非中文 locale 下隐藏或标注为 "Chinese Lunar Calendar"。
- R17.6 习惯模板和推荐目标应按语言提供对应版本。

**验收标准**

- 系统语言为英文时，首次启动多仪，界面显示为英文。
- 在偏好设置中切换语言后，应用全部文案即时更新，无需重启。
- 中文环境下日期显示「5月15日 周四」，英文环境下显示「May 15, Thu」。
- 时区从 Asia/Shanghai 切换到 America/New_York 后，已设置的提醒时间自动调整。

## 5. 非功能要求

- 离线优先：网络不可用时可以继续新增、编辑、完成。
- 幂等：同步、调度、重放不产生重复记录。
- 可诊断：通知失败、同步失败、权限不足有可见状态。
- 可迁移：模型版本升级必须向后兼容。
- 可测试：核心规则不依赖平台插件即可测试。
- **[v2 新增]** 可访问：所有新增 UI 组件应提供 Semantics 标注，支持 TalkBack/VoiceOver。
- **[v2 新增]** 性能：看板视图在 200 条待办下拖拽帧率不低于 50fps。
- **[v2 新增]** 包体积：新增白噪音/场景音应使用压缩格式，单个音频文件不超过 2MB。
