# 多仪需求规格 v2

本文档在 v1（R1-R10）基础上，更新各需求的实现状态，并基于竞品对标分析
新增 R11-R17。目标是将多仪从「功能完备的个人效率工具」推向「智能化、社交化、
国际化的全平台时间管理系统」。

---

## 1. 背景

### 1.1 当前已有能力（截至 v1.0.21）

- 待办：四象限、清单、子任务、重复、截止日、多次提醒（ReminderPlan/ReminderRule）、优先级、标签、AI 拆解子任务。

### 1.3 v2 新增对标缺口（竞品分析）

基于对滴答清单、Forest、Habitica、Notion、日事清等产品的竞品分析，新增：

- 智能日期解析：中英文自然语言输入自动解析日期/时间/提醒（对标滴答清单）。
- 智能日程建议：根据截止日、优先级、四象限和任务日期推荐今日应做任务，并支持一键加入今日（对标 Microsoft To Do My Day）。**[已实现]**
- 看板视图：除四象限外，支持看板拖拽管理任务（对标 Notion/Trello）。
- 专注模式增强：DND 联动、专注社交、更丰富的场景音、标签分类统计（对标 Forest）。
- 习惯追踪增强：模板库扩展、分组分类、弹性打卡（对标 Habitica/Streaks）。
- 数据可视化增强：年度报告、趋势分析、多维交叉分析（对标日事清/滴答清单）。
- 快捷操作：通知栏快捷添加、小组件快捷添加、快捷模板（对标各主流效率应用）。
- 国际化支持：多语言、时区自动处理（面向海外用户拓展）。

## 2. 产品目标

1. 多仪应成为一个「可落地执行」的个人时间系统，而不是只记录任务。
2. 所有提醒必须可验证、可诊断、可重新调度。
3. 所有跨模块事项必须能在日历中统一查看、跳转、编辑。
4. 核心数据必须支持多人共享和冲突处理，同时保留离线可用。
5. 时间统计必须来自真实时间记录，而不是只用完成数量推断。
6. **[v2 新增]** 输入效率：自然语言和快捷操作应让「从想法到记录」的路径尽可能短。
7. **[v2 新增]** 视觉洞察：数据可视化应帮助用户理解自己的时间使用模式和长期趋势。
8. **[v2 新增]** 国际化：应用应对中文和英文用户提供一致体验。

### R1 日历枢纽

> **v2 状态：已部分实现。** CalendarAggregator 已聚合全部 9 种事件类型（todo、
> habit、pomodoro、anniversary、course、diary、countdown、goal、timeEntry），
> 已实现冲突检测和标记（hasConflict/conflictCount），CalendarEventSheet 已提供
> 统一详情面板。日历详情已支持本地日程编辑/删除，待办完成/改期/调整时间/删除，
> 目标/纪念日/倒数日改期/删除，纪念日、倒数日、课程、日记和习惯复用原模块
> 编辑器直接回写，时间足迹改期/调整开始/调整时长/删除，番茄钟会话已支持编辑
> 并同步时间足迹。尚需继续回归更多真机路径与残余平台差异。

**用户故事**

用户希望在日历里看到今天所有待办、习惯、课程、目标、倒数日、纪念日和日记，
并能直接处理，不需要在多个模块间来回找。

**功能要求**

- R1.1 日历应聚合所有模块事件，统一为 `CalendarEvent` 展示。 **[已实现]**
- R1.2 日/周/月视图都应支持按类型筛选。 **[部分实现]**
- R1.3 点击任一事件应打开统一详情面板，提供跳转、完成、改期、编辑、删除等动作。 **[部分实现 - 本地日程/待办/目标/纪念日/倒数日/课程/日记/习惯/时间足迹/番茄钟已有回写动作]**
- R1.4 当同一时间段存在课程、待办提醒、目标提醒时，应提示冲突。 **[已实现]**
- R1.5 今日页和日历页应共享同一套事件排序规则。 **[已实现]**
- R1.6 修改日历中的截止日/提醒时间后，应同步写回原模块 provider。 **[部分实现 - 待办、目标、纪念日、倒数日、课程、日记、习惯、时间足迹、番茄钟、本地日程已写回]**

**验收标准**

- 新建一条带截止日期和提醒的待办后，日历日视图、周视图、月视图均可见。
- 在日历中改期后，待办详情页显示新日期。
- 删除原待办后，日历事件消失。
- 冲突事件至少以视觉标记和详情提示展示。

### R2 多次提醒与稍后提醒

> **v2 状态：已基本实现。** ReminderPlan + ReminderRule 模型已落地，支持
> absolute/relativeToDue/dailyTime/weeklyTime 四种规则类型，push/alarm 双通道
> 路由已完成，ReminderScheduler 实现幂等重放，legacy 提醒自动迁移。
> 稍后提醒（snooze）已接入通知 action 和 deep-link 快捷路径，真机端到端回归仍需继续确认。

**用户故事**

用户希望一个任务可以提前一天、提前一小时、到期时分别提醒；错过后可以稍后再提醒。

**功能要求**

- R2.1 `ReminderConfig` 应支持多条 `ReminderRule`。 **[已实现 - ReminderPlan 含 rules 列表]**
- R2.2 ReminderRule 支持绝对时间、相对截止时间、每日固定时间、周几固定时间。 **[已实现 - 4 种 ReminderRuleType]**
- R2.3 每条 rule 可选择 push 或 alarm。 **[已实现 - ReminderKind.push/alarm]**
- R2.4 闹钟提醒支持全屏、震动、稍后提醒。 **[已实现代码链路 - 通知 action / deep-link 已接入，待真机端到端回归]**
- R2.5 提醒调度必须可幂等重放，避免重复注册。 **[已实现 - ReminderScheduler.resyncAll]**
- R2.6 切换 push/alarm 后必须清理旧通道残留。 **[已实现 - 双通道清理逻辑]**
- R2.7 通知点击应能跳转到对应任务/目标/习惯详情，而不仅是 Tab。 **[已实现 - duoyi:// 深链]**
- R2.8 失败的单条提醒不得中断整轮调度。 **[已实现 - try/catch 隔离]**
- R2.9 位置提醒支持到达/离开触发，并可从任务详情创建关联提醒。 **[已实现 - LocationReminderEngine + TodoDetailScreen 任务关联 + Android 系统 geofence 调度 + 前台/后台位置授权引导；OEM 行为待真机验证]**

**验收标准**

- 一个待办设置 3 条提醒后，pending notification 中存在 3 条可追踪 id。
- 修改其中一条规则后，只保留最新规则对应的调度。
- 关闭通知权限后 UI 显示异常状态；重新授权回到前台后自动刷新并重排。
- Android 精准闹钟权限缺失时，至少退化为普通提醒，并在 UI 引导用户修复。

## 5. 非功能要求

- 离线优先：网络不可用时可以继续新增、编辑、完成。
- 幂等：同步、调度、重放不产生重复记录。
- 可诊断：通知失败、同步失败、权限不足有可见状态。
- 可迁移：模型版本升级必须向后兼容。
- 可测试：核心规则不依赖平台插件即可测试。
- **[v2 新增]** 可访问：所有新增 UI 组件应提供 Semantics 标注，支持 TalkBack/VoiceOver。
- **[v2 新增]** 性能：看板视图在 200 条待办下拖拽帧率不低于 50fps。
- **[v2 新增]** 包体积：新增白噪音/场景音应使用压缩格式，单个音频文件不超过 2MB。
