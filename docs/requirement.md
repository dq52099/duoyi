# 多仪对标指尖时光需求规格

本文档定义多仪下一阶段的产品需求。目标不是复制竞品，而是在现有
「待办 / 习惯 / 目标 / 日历 / 番茄 / 课程表 / 倒数日 / 云同步」基础上，
补齐公开可见的成熟日程产品能力：共享协作、多次提醒、时间足迹、提醒可靠性、
日历枢纽和成就闭环。

## 1. 背景

### 1.1 当前已有能力

- 待办：四象限、清单、子任务、重复、截止日、提醒、优先级、标签。
- 习惯：打卡、连续天数、热力图、目标次数、提醒。
- 目标：推荐目标、里程碑、提醒、进度。
- 日历：月/周/日视图，聚合事件。
- 专注：番茄钟、白噪音、会话历史。
- 个人中心：统计、成就、公告、反馈、主题、偏好、通知权限、检查更新、云同步。
- 平台：Android 通知/闹钟、桌面通知、桌面托盘、Android 小组件、GitHub Release 更新。

### 1.2 对标后确认的主要缺口

- 共享协作：缺少好友共享日程、共享清单、权限和协同记录。
- 多次提醒：缺少一个事项绑定多条提醒、提前提醒、稍后提醒、重复响铃。
- 时间足迹：缺少活动时间流水、手动补记、自动记录、时间审计报表。
- 日历枢纽：已有聚合，但缺少统一的日程详情、冲突提示、快速改期和跨模块编辑。
- 成就闭环：已有成就页，但缺少事件驱动的徽章、目标奖励和可见反馈。
- 手机通知可靠性：已有权限页，但还要补小米等系统的自启动、后台、锁屏、渠道健康检查。
- 小组件联动：已有基础小组件，但还要补更多尺寸、设置入口和事项完成动作。

## 2. 产品目标

1. 多仪应成为一个「可落地执行」的个人时间系统，而不是只记录任务。
2. 所有提醒必须可验证、可诊断、可重新调度。
3. 所有跨模块事项必须能在日历中统一查看、跳转、编辑。
4. 核心数据必须支持多人共享和冲突处理，同时保留离线可用。
5. 时间统计必须来自真实时间记录，而不是只用完成数量推断。

## 3. 范围

### 3.1 本期范围

- 多次提醒模型和调度。
- 通知健康检查与系统设置引导。
- 时间足迹模型、记录页、统计接入。
- 日历统一事件详情与快速操作。
- 共享日程/共享清单的后端合同与前端最小可用实现。
- 成就事件管线和奖励反馈。
- 课程表、倒数日、纪念日与日历/提醒的深度联动。

### 3.2 暂不进入本期

- 社交动态流。
- 第三方日历双向同步。
- 付费订阅、会员权益、应用商店内购。
- 复杂团队项目管理。

## 4. 需求

### R1 日历枢纽

**用户故事**

用户希望在日历里看到今天所有待办、习惯、课程、目标、倒数日、纪念日和日记，
并能直接处理，不需要在多个模块间来回找。

**功能要求**

- R1.1 日历应聚合所有模块事件，统一为 `CalendarEvent` 展示。
- R1.2 日/周/月视图都应支持按类型筛选。
- R1.3 点击任一事件应打开统一详情面板，提供跳转、完成、改期、编辑、删除等动作。
- R1.4 当同一时间段存在课程、待办提醒、目标提醒时，应提示冲突。
- R1.5 今日页和日历页应共享同一套事件排序规则。
- R1.6 修改日历中的截止日/提醒时间后，应同步写回原模块 provider。

**验收标准**

- 新建一条带截止日期和提醒的待办后，日历日视图、周视图、月视图均可见。
- 在日历中改期后，待办详情页显示新日期。
- 删除原待办后，日历事件消失。
- 冲突事件至少以视觉标记和详情提示展示。

### R2 多次提醒与稍后提醒

**用户故事**

用户希望一个任务可以提前一天、提前一小时、到期时分别提醒；错过后可以稍后再提醒。

**功能要求**

- R2.1 `ReminderConfig` 应支持多条 `ReminderRule`。
- R2.2 ReminderRule 支持绝对时间、相对截止时间、每日固定时间、周几固定时间。
- R2.3 每条 rule 可选择 push 或 alarm。
- R2.4 闹钟提醒支持全屏、震动、稍后提醒。
- R2.5 提醒调度必须可幂等重放，避免重复注册。
- R2.6 切换 push/alarm 后必须清理旧通道残留。
- R2.7 通知点击应能跳转到对应任务/目标/习惯详情，而不仅是 Tab。
- R2.8 失败的单条提醒不得中断整轮调度。

**验收标准**

- 一个待办设置 3 条提醒后，pending notification 中存在 3 条可追踪 id。
- 修改其中一条规则后，只保留最新规则对应的调度。
- 关闭通知权限后 UI 显示异常状态；重新授权回到前台后自动刷新并重排。
- Android 精准闹钟权限缺失时，至少退化为普通提醒，并在 UI 引导用户修复。

### R3 手机通知健康检查

**用户故事**

小米手机用户开启提醒后仍收不到通知，希望应用能告诉他还缺哪些系统开关。

**功能要求**

- R3.1 偏好设置提供通知健康检查页。
- R3.2 检查系统通知权限、精准闹钟权限、通知渠道状态。
- R3.3 Android 上提供「系统应用设置」跳转。
- R3.4 小米/MIUI 设备显示自启动、后台限制、锁屏通知、电池优化的人工检查清单。
- R3.5 发送测试通知后记录结果和时间。
- R3.6 从系统设置返回时刷新权限并重排提醒。

**验收标准**

- 未授权通知时，提醒编辑页显示未授权并提供设置跳转。
- 授权后返回应用，状态自动变为已授权。
- 小米设备显示专门的配置说明。
- 测试通知触发后，通知记录页新增一条记录。

### R4 时间足迹与时间审计

**用户故事**

用户希望知道自己一天时间花在哪些事情上，而不是只知道完成了几个任务。

**功能要求**

- R4.1 新增 `TimeEntry`：开始、结束、时长、类型、关联对象、备注、来源。
- R4.2 番茄完成后自动生成专注 TimeEntry。
- R4.3 待办完成时可选择记录耗时。
- R4.4 习惯打卡可按目标单位换算时间，或手动补记。
- R4.5 日历页支持添加一段活动时间。
- R4.6 统计页新增时间分布、时间线、周/月趋势、模块占比。
- R4.7 支持手动编辑和删除时间记录。

**验收标准**

- 完成一次 25 分钟番茄后，时间足迹出现一条 25 分钟记录。
- 手动补记 30 分钟阅读后，统计页阅读类时间增加。
- 删除记录后统计同步回退。
- 时间记录参与云同步。

### R5 共享日程与协作

**用户故事**

用户希望把一个清单或日程共享给朋友/家人，对方可查看或共同编辑。

**功能要求**

- R5.1 新增共享空间 `Workspace`，个人数据默认在 private workspace。
- R5.2 支持共享清单、共享日程、共享目标。
- R5.3 成员角色：owner、editor、viewer。
- R5.4 邀请方式：邀请码或链接码。
- R5.5 支持撤销成员、修改角色。
- R5.6 同步时按 workspace 合并数据。
- R5.7 冲突策略：同字段 last-write-wins，并保留变更记录。
- R5.8 UI 应显示共享标识和成员头像/昵称。

**验收标准**

- A 创建共享清单并邀请 B，B 登录后可看到该清单。
- viewer 无法编辑，editor 可新增待办。
- A 移除 B 后，B 不能继续同步该共享空间。
- 离线编辑后恢复网络，冲突结果可解释。

### R6 成就与目标反馈

**用户故事**

用户希望完成目标、保持连续习惯、专注达成时能得到明确奖励。

**功能要求**

- R6.1 建立成就事件管线：todo_completed、habit_streak、focus_minutes、goal_milestone。
- R6.2 成就规则应数据驱动，便于新增。
- R6.3 解锁成就后显示轻量反馈，并写入通知/历史。
- R6.4 成就墙展示未解锁、已解锁、进度。
- R6.5 目标里程碑完成时触发奖励。

**验收标准**

- 连续打卡 7 天后解锁对应徽章。
- 解锁弹层不会打断保存或计时流程。
- 成就进度在重启后保持。

### R7 课程表、倒数日、纪念日深度联动

**功能要求**

- R7.1 课程表支持学期范围、单双周、节次时间、地点、老师。
- R7.2 课程事件进入日历和今日页。
- R7.3 倒数日支持提醒、置顶、分类。
- R7.4 纪念日支持农历/公历、提前提醒。
- R7.5 所有这些事件都可以通过统一详情面板跳回原页面。

**验收标准**

- 当前周课程在周视图准确出现。
- 单双周课程只在对应周出现。
- 倒数日提醒能被 ReminderScheduler 管理。

### R8 小组件与深链

**功能要求**

- R8.1 小组件支持待办完成、开始番茄、今日事件列表。
- R8.2 点击小组件应深链到具体对象详情。
- R8.3 小组件刷新不应阻塞主应用启动。
- R8.4 数据变化后应推送最新摘要到小组件。

**验收标准**

- 点击小组件中的待办直接打开该待办详情。
- 完成待办后小组件数量更新。

### R9 同步、备份与隐私

**功能要求**

- R9.1 新模型必须进入云同步 payload。
- R9.2 旧数据自动迁移，不丢失提醒。
- R9.3 导出备份包含所有新增模型。
- R9.4 用户可删除共享空间和本地时间记录。
- R9.5 不上传系统权限状态、设备设置等本机隐私数据。

**验收标准**

- 老版本数据升级后能正常启动。
- 备份导入后时间足迹、提醒规则、共享元数据完整。

### R10 质量要求

- R10.1 所有模型新增 JSON 兼容测试。
- R10.2 提醒调度必须有属性测试覆盖。
- R10.3 关键页面有 widget 测试。
- R10.4 Android 通知相关流程提供手工回归清单。
- R10.5 `flutter analyze` 不得出现 error。
- R10.6 新增 UI 必须使用现有 `surface_components.dart` 和设计 token。

## 5. 非功能要求

- 离线优先：网络不可用时可以继续新增、编辑、完成。
- 幂等：同步、调度、重放不产生重复记录。
- 可诊断：通知失败、同步失败、权限不足有可见状态。
- 可迁移：模型版本升级必须向后兼容。
- 可测试：核心规则不依赖平台插件即可测试。

## 6. 优先级

- P0：通知健康、权限刷新、系统设置跳转。
- P1：多次提醒、提醒重放、深链到详情。
- P2：时间足迹、统计接入。
- P3：日历统一详情和跨模块编辑。
- P4：共享日程/清单。
- P5：成就事件管线。
- P6：小组件增强和课程表/倒数日深度联动。
